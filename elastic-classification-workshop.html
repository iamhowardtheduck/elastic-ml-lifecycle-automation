Chapter 1
01

Build & Train a
Classification Job

Create, train, and deploy the mortgage-privileged-access-classification job — predict whether a privileged access event is LOW, MEDIUM, or HIGH risk by learning from PingOne IAM and Oracle DB audit events simultaneously.
Chapter 1 — Step 1.1

Explore the Source Data

logs-ping_one.audit-mortgage
PingOne IAM Audit Events
Add columns:
• ping_one.audit.risk.level — the label (LOW/MED/HIGH)
• ping_one.audit.risk.score — numeric 0–100
• ping_one.audit.action.type — IAM action
• ping_one.audit.result.status — SUCCESS/FAILED/BLOCKED
• event.outcome — success/failure

Notice: LOW events dominate, HIGH events are rare — this is the class imbalance the maximize_minimum_recall objective addresses.
logs-oracle.database_audit-mortgage
Oracle DB Audit Events
Add columns:
• oracle.database_audit.action — SELECT, DROP TABLE, GRANT
• oracle.database_audit.privilege — SYSDBA, DBA, NOT APPLICABLE
• event.outcome — success/failure

Notice: No PingOne fields present — DFA handles null features naturally.

Discussion: Why train one model on both sources rather than two separate models?

A single cross-source model sees the full feature space. It learns that SYSDBA privilege escalation in Oracle carries the same risk signature as USER.MFA.BYPASS in PingOne — because both co-occurred with HIGH risk labels in training. Two separate models only see half the picture each.

Set time picker to Last 30 days before exploring. The data generator has embedded diurnal patterns — you should see peaks at 10:00–12:00 local time and near-zero volume overnight.

Chapter 1 — Step 1.2

Check Class Balance with ES|QL

-- Class distribution: PingOne source
FROM logs-ping_one.audit-mortgage
| WHERE @timestamp > NOW() - 30 days
| STATS doc_count = COUNT(*)
        BY ping_one.audit.risk.level
| SORT doc_count DESC
-- Verify Oracle source + high-risk signals
FROM logs-oracle.database_audit-mortgage
| WHERE @timestamp > NOW() - 30 days
| STATS
  total       = COUNT(*),
  sysdba_count = SUM(CASE(privilege ==
                    "SYSDBA", 1, 0)),
  drop_count   = SUM(CASE(action ==
                    "DROP TABLE", 1, 0))

What you expect to see: Three rows — LOW dominating, MEDIUM moderate, HIGH rarest. You need several hundred in each class for reliable training.

ES|QL tip: COUNT(field) counts non-null rows — it does NOT filter on a condition. Use SUM(CASE(condition, 1, 0)) to count rows matching a condition. The workshop queries use this pattern throughout.

Why this matters: If HIGH is very sparse, the maximize_minimum_recall objective you'll set in Step 1.3 specifically penalises the model for missing rare classes — it's the right choice for imbalanced security data.

Why the wizard first? It generates the base JSON structure — source indices, dependent variable, training_percent — which you retrieve and extend via Dev Tools rather than writing from scratch.

The key gap: Without an excludes list, DFA would analyze @timestamp, user.id, and user.name. The model would memorize when events happen and who performed them — not the behavioral patterns that signal genuine risk.

Chapter 1 — Step 1.3 Part B

Recreate with Excludes + Hyperparameters

# First — retrieve what the wizard created
GET _ml/data_frame/analytics/
  mortgage-privileged-access-classification

# Delete the wizard job
DELETE _ml/data_frame/analytics/
  mortgage-privileged-access-classification

# Recreate with correct configuration
PUT _ml/data_frame/analytics/
  mortgage-privileged-access-classification
{
 "source": { "index": ["logs-ping_one.audit-mortgage",
                "logs-oracle.database_audit-mortgage"] },
 "dest": { "index": "mortgage-privileged-access-classification",
           "results_field": "ml" },
 "analysis": { "classification": {
   "dependent_variable": "ping_one.audit.risk.level",
   "training_percent": 80, "num_top_classes": 3,
   "prediction_field_name": "risk_level_prediction",
   "num_top_feature_importance_values": 5,
   "class_assignment_objective": "maximize_minimum_recall",
   "lambda": 0.1, "gamma": 0.1,
   "eta": 0.1, "feature_bag_fraction": 0.5 }},
 "analyzed_fields": {
   "includes": ["ping_one.audit.risk.score",
     "ping_one.audit.risk.level", "ping_one.audit.action.type",
     "ping_one.audit.result.status",
     "oracle.database_audit.action",
     "oracle.database_audit.privilege", "event.outcome"],
   "excludes": ["@timestamp", "user.id", "user.name"] },
 "model_memory_limit": "550mb"
}

3 changes from the wizard:

① analyzed_fields.excludes populated
@timestamp, user.id, user.name stripped from analysis. Prevents the model from learning time-based or identity-based patterns instead of behavioral risk signals.
② Hyperparameters added as flat fields
lambda, gamma, eta, feature_bag_fraction are siblings of dependent_variable — not nested under a "hyperparameters" key (the API rejects that).
③ Tight includes list
Only the 7 fields that carry genuine risk signal. Keeps the feature space clean and destination index lean.

Chapter 1 — Step 1.4

Start the Job & Monitor Training

Navigate to Data Frame Analytics → Jobs. Click ▶ Start on the mortgage-privileged-access-classification row. Press Refresh to watch phases advance.

PhaseWhat Happens
reindexingAll documents from both sources copied to destination index
loading_dataFeatures extracted and prepared for boosted tree algorithm
feature_selectionZero-variance fields dropped before training
coarse_parameter_searchBayesian search over hyperparameter space — slowest phase
fine_tuning_parametersCoarse candidates narrowed to final values
final_trainingModel trained on 80% of data; hyperparameters locked
writing_resultsPredictions written to held-out 20% test set
inferencePrecision, recall, confusion matrix computed

Training time: 3–8 minutes depending on document count and cluster size. Watch the progress bar in Data Frame Analytics → Jobs.

While it trains — discuss: DFA is learning on a dataset where roughly half the documents have null PingOne fields and half have null Oracle fields. The model learns that SYSDBA privilege + failed outcome from Oracle carries the same HIGH-risk signature as USER.MFA.BYPASS from PingOne — cross-source pattern recognition that rule-based systems cannot do.

After completion: Click Synchronize your jobs and trained models before exploring results. This syncs the trained model with Kibana's model registry.

Chapter 1 — Step 1.5

Explore Training Results

Confusion Matrix
Click Explore results on the completed job. Look for:

• HIGH recall — what % of actual HIGH events predicted correctly? Because you used maximize_minimum_recall this should be strongest.
• LOW precision — how many LOW predictions were actually LOW?
• Any systematic MEDIUM↔HIGH misclassification?
Feature Importance
Click any document in the results table. The Feature Importance panel shows which fields drove that specific prediction.

For a HIGH-risk document you should see ping_one.audit.risk.score or oracle.database_audit.privilege near the top.

Discussion: Feature importance answers why the model predicted — not just what. In mortgage compliance, showing an auditor why a transaction was flagged is as important as flagging it correctly.

Chapter 1 — Step 1.8

Wire Pipeline to Both Data Streams

# Apply pipeline at data stream level
# Propagates to current backing index
# + all future rollovers automatically

PUT /logs-ping_one.audit-mortgage/_settings
{
 "index": {
  "default_pipeline": "mortgage-privileged-access-ml"
 }
}

PUT /logs-oracle.database_audit-mortgage/_settings
{
 "index": {
  "default_pipeline": "mortgage-privileged-access-ml"
 }
}

# Verify both settings applied
GET /logs-ping_one.audit-mortgage/_settings/index.default_pipeline
GET /logs-oracle.database_audit-mortgage/_settings/index.default_pipeline

Why data stream level? Applying _settings to the data stream name rather than the backing index (.ds-logs-*) means the pipeline propagates to the current index and all future rollovers without re-applying manually.

Optional: View in UI — Search ingest pipelines → find mortgage-privileged-access-ml → click Edit pipeline → click the Inference processor to see the full model configuration in the UI.

Troubleshooting: If new documents show ml.inference_error, the model ID in the pipeline may not match the actual trained model. Run:
GET _ml/trained_models?tags=mortgage-privileged-access-classification and compare.

ML Readiness Analyst & Automation Workflow

Steps 2.1 – 2.3
Build the ML Readiness Analyst Agent
Enable Workflows in Advanced Settings. Build the agent using built-in platform tools only:

① list_indices — mandatory first action
② Schema discovery — field types and mappings
③ Class balance check — ES|QL histogram on audit.is_suspicious
④ Feature analysis — correlation with label field
⑤ Readiness verdict + recommended hyperparameters
⑥ Trigger the Automation Workflow
Steps 2.4 – 2.7
Build & Run the Automation Workflow
Parameterised workflow receives the agent's schema findings:

① Check data readiness (document count, field existence)
② Create DFA classification job via API
③ Start job and poll training phases to completion
④ Create inference pipeline for the trained model
⑤ Apply pipeline to logs-mortgage.audit-default
⑥ Review Workflow execution log

The agent's job = understand the data. The Workflow's job = act on that understanding. Together they reduce a multi-hour manual process — the same steps you performed in Chapter 1 — to a fully automated pipeline that runs without human intervention.

Chapter 3 — Take-Aways

Key Takeaways

Excludes matter as much as includes
The Kibana wizard only exposes includes. Explicitly excluding @timestamp, user.id, and user.name is what makes the model learn behavioral risk patterns rather than time-of-day patterns or individual user habits.
Cross-source training unifies risk language
A single model trained on PingOne + Oracle normalizes risk across two completely different systems. The downstream pipeline produces ml.risk_level_prediction = HIGH regardless of where the event originated.
Ingest pipelines make ML operational
Every new event is scored at write time. No batch jobs, no separate inference service, no latency. The model output lives alongside the raw data in the same index — queryable, alertable, dashboardable.
